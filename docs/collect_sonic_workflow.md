# collect_sonic_demos 数据采集工作流

该文档在《RoboCasa Agent 工程工作流》的基础上，专门针对数据采集脚本 `collect_sonic_demos.py` 的开发与优化，帮助团队更便捷地在仿真中采集机器人演示数据。

## 1. 目标

- 支持人形机器人或其它自定义机器人在 SONIC 控制器下收集演示数据。
- 将采集流程标准化，易于复用和扩展。
- 保证采集的数据格式兼容现有 RoboCasa365 数据集和下游学习算法。

## 2. 角色与分工

### 数据采集脚本优化代理

- **建议模型**：`GPT-5.3 Spark / medium-high`
- **职责**：
  1. 阅读并分析 `collect_sonic_demos.py` 脚本的现有功能与接口。
  2. 依据需求撰写数据采集计划，明确环境、场景、机器人配置、数据格式和 CLI 设计。
  3. 实现或修改脚本，包括添加所需参数、适配人形机器人模型、接入相机/传感器数据保存等。
  4. 编写最小验证脚本或测试，确保采集流程可用且数据正确保存。

- **输出**：
  - `data_collection_plan.md`：采集设计与任务拆解。
  - `script_diff`：对 `collect_sonic_demos.py` 的代码修改。
  - `data_collection_notes.md`：采集过程记录、测试命令和结果。

- **禁止事项**：
  - 修改与数据采集无关的其它脚本或模块。
  - 将硬件依赖代码混入仿真脚本。
  - 在未更新计划或未通过审核前，自行扩大范围。

## 3. 建议的开发流程

1. **需求冻结**：由总控代理完成需求文档，阐述为什么需要优化数据采集脚本以及采集目标（比如人形机器人、任务类型、数据量）。
2. **仓库分析**：仓库结构代理标出当前脚本所在路径（例如 `robocasa/scripts/collect_sonic_demos.py`）及可能复用的工具（如已有的 data collection wrapper、dataset conversion 脚本等）。
3. **采集计划**：数据采集脚本优化代理填写 `data_collection_plan.md`，细化改进点、设计方案和任务拆解。
4. **实现与测试**：实现代理根据计划修改脚本，新增适当的测试用例或 smoke test，记录在 `implementation_notes.md` 和 `test_log.md`。
5. **功能与代码评审**：功能评审代理和代码审查代理按标准流程验收采集脚本是否达标。
6. **合并与发布**：总控代理在所有 gate 通过后，建议合并 PR。之后可以使用新脚本批量生成演示数据。

## 4. 采集脚本结构建议

采集脚本一般包括以下部分：

- **参数解析**：读取任务名、机器人型号、场景种类、采集数量、随机种子等输入。
- **环境创建**：使用 `gymnasium.make` 创建 RoboCasa 环境，设置 `split`、`seed` 等。
- **控制循环**：加载 SONIC 控制器或其它控制策略，对机器人进行动作控制。
- **数据记录**：在每个时间步保存观测、动作和任务进度信息，建议保存为 `npz` 或 `hdf5` 格式。
- **可选功能**：通过 `--save-images` 保存渲染图像，通过 `--save-meta` 保存奖励和终止标记，并可使用 `--camera-name`、`--camera-width`、`--camera-height` 设置图像渲染参数。
- **结束与清理**：达到终止条件或固定 horizon 后保存文件并释放资源。

## 5. 风险与注意事项

- 仿真环境和控制器可能会消耗大量 CPU/GPU 资源，建议在调试时使用较小 horizon、seed 或任务数量。
- 数据写入路径应当唯一，避免覆盖已有演示。
- 对人形机器人的动态稳定性进行评估，避免控制器发散导致无效数据。
